# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

Read and follow `AGENTS.md` before working. Treat it as the shared cross-provider source of truth.

## Project overview

"Our Story — The Missing Flight" is a mobile-first, cooperative 2.5D point-and-click puzzle adventure delivered as a static PWA. There is no build step, bundler, or npm dependency — plain ES modules served statically.

## Commands

```bash
npm test                        # = node tests/run-tests.mjs (exits non-zero on failure)
npm start                       # = python3 -m http.server 8080, then open http://localhost:8080
```

There is a single test file (`tests/run-tests.mjs`); there is no way to run one test in isolation. Before completion also run `git diff --check` and `git status --short`.

The test suite validates: scene/chapter graph integrity, puzzle validators, the clue dependency graph (no orphan clues, no missable mandatory clues), cutscenes, save migrations, every referenced PNG's existence — and it fails on any SVG/PDF/WebP file or reference, or on any plaintext occurrence of the mystery's final answer.

## Architecture

- **Entry point**: `index.html` loads only `scripts/engine.js` (as an ES module, with a `?v=` cache-busting query). Root-level `game.js` and `sprite-loader.js` are legacy Chapter-1 prototype code not referenced by `index.html`.
- **`scripts/engine.js`**: boot, JSON data loading, the global `game` object (data, save, mode, render/persist/navigation), and panel wiring. All other modules receive `game` as a parameter.
- **`scripts/core.js` and `scripts/save-core.js`**: pure logic modules with no DOM or localStorage — they are imported directly by the Node test suite. Keep them pure. `core.js` holds answer normalization/checking (including SHA-256 hash validation for the final answer), unlock-condition parsing, and scoring. `save-core.js` holds the versioned save schema (`SAVE_VERSION`), migrations, and (de)serialization.
- **Browser-only modules**: `scene.js` (exploration/hotspots), `puzzle.js`, `cutscene.js`, `mystery.js` (notebook/passport/route/inventory panels), `dialogue.js`, `home.js` (title/save slots), `save.js` (localStorage wrapper around save-core), `ui.js` (modal/toast/DOM helpers).
- **Data-driven content**: all game content lives in `data/*.json` — `chapters`, `scenes`, `puzzles`, `cutscenes`, `mystery`, `characters` (the `DATA_FILES` list in engine.js). Content changes usually mean editing these JSON files, not code; the test suite cross-validates them.
- **Spoiler rule**: the mystery's final answer is never stored in plaintext anywhere in the repo — it is validated by SHA-256 hash (`validator: 'hash'` in puzzles). Tests enforce this.
- **PWA**: `sw.js` (network-first cache; bump the `CACHE` constant when changing cached behavior) + `manifest.webmanifest`.

## Assets

- **PNG only** — SVG, PDF, and WebP are prohibited and enforced by tests (`docs/PNG_ONLY_RULE.md`). Preserve exact filename casing and all existing PNG references. Preserve transparent alpha for isolated sprites.
- Character art comes from the approved transparent PNG atlases in `assets/atlases/` (see `docs/APPROVED_CHARACTER_SPRITES.md` and `docs/FINAL_CHARACTER_ASSETS.md`; do not resurrect old character art).
- Placeholders are generated by `tools/generate_placeholders.py`; the art brief for every file lives in `assets/ASSET_TEMPLATE.json`. Replace placeholders with painted art at the same path.
- Character consistency: Leah — headscarf, long sleeves, long skirt in every sprite; Moshé — curly dark hair, short beard, visible kippah.

## Authority and model routing

- Human user: final authority.
- Fable 5: lead architect, task decomposer, disagreement resolver, and final reviewer.
- Sonnet: primary implementation, substantial debugging, refactoring, UI/state work, test creation, and asset-system integration.
- Haiku: repository exploration, inventories, searches, logs, test execution, path checks, diff summaries, and Codex handoff preparation.
- Codex: independent implementation or review through a real Codex task, branch, PR, issue, action, or committed handoff.

Fable should normally orchestrate at low effort. Increase effort for ambiguous architecture, difficult root-cause analysis, security-sensitive changes, cross-cutting work, conflicting reports, and final approval.

Do not spend Fable on routine searches, repetitive edits, or ordinary test logs when Haiku or Sonnet can handle them.

## Project agents

- `repo-explorer`: read-only investigation using Haiku.
- `code-worker`: scoped implementation using Sonnet.
- `asset-worker`: visual-asset and asset-system work using Sonnet.
- `test-runner`: read-only validation using Haiku.
- `codex-handoff`: creates structured Codex handoff documents using Haiku.
- `final-reviewer`: independent final review using Fable.

## Required substantial-task workflow

1. Inspect git status and repository instructions.
2. Use `repo-explorer` for evidence-based discovery.
3. Create a concrete plan.
4. Decide whether Claude alone, Codex alone, competing implementations, or cross-review is appropriate.
5. Delegate substantial Claude implementation to Sonnet.
6. Run Haiku validation.
7. When Codex is needed, create a handoff under `docs/agent-handoffs/`.
8. Never claim Codex was invoked unless an actual Codex task or integration was used.
9. Inspect Codex's actual diff when it returns.
10. Compare results and delegate concrete corrections to Sonnet.
11. Run `test-runner` and then `final-reviewer`.
12. Reject completion while blocking findings remain.
13. Report exact files, commands, results, limitations, and final approval status.

Do not allow agents to edit overlapping files concurrently. Do not commit, push, merge, deploy, publish, or open a pull request unless explicitly requested.

## Engineering rules (from AGENTS.md)

- Mobile first. Preserve unrelated behavior and existing user changes.
- Reuse current architecture before adding new systems; prefer targeted fixes over broad rewrites.
- Do not add dependencies without a concrete need.
- Serialize tightly coupled code, loader, manifest, atlas, and service-worker changes; parallelize only independent read-heavy tasks.
- Clearly separate pre-existing failures from newly introduced failures.
